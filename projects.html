Raft K-V

Python | Individual Project

Feb 2020 - May 2020

A more robust attempt of the Raft consensus protocol compared to my first attempt in 2017, this time building a key-value store on top of it as well. This implements the entire protocol excluding cluster membership change and snapshotting. I detail interesting design decisions for the key components:

  • Leader Election: The challenge associated with leader election and maintaining authority for the leader arose due to the abundance of threads that needed to be spawned - the timeout threads for followers and candidates, and the heartbeat thread for the leader. I had to be meticulous about cancelling threads so that for instance, followers wouldn’t start an election despite hearing from the leader. Multithreaded servers also necessitated the use of locks to preserve consistency. Locks introduced overhead and programming complexity, as I reasoned about the criticial sections that modified shared server state between the threads. I also ignored (delayed) stale RPCs from older terms upon receiving them.
  • Log Replication: Log replication was the most difficult part to implement because of edge cases. Since heartbeats from the leader to followers are sent via AppendEntriesRPCs and could be sent while the leader’s log is still empty, I needed a clean way for a follower to differentiate between receiving such a heartbeat, and receiving an AppendEntriesRPC adding new entries to an empty log. My solution was to factor out the case where the leader was sending a heartbeat with an empty log; then, following the rest of the Raft Figure 2 would take care of remaining inconsistencies.
    The Raft paper states that if commitIndex > lastApplied at a server, then apply the commands in the log between lastApplied and the commitIndex. Because of potential race conditions with multiple threads and to prevent blocking the server from concurrently receiving messages by applying commands with the main thread, I opted to use one dedicated thread to apply commands. This thread would wait on a condition variable that is only notified at the leader when it receives success responses for AppendEntriesRPCs and at a follower when it receives an AppendEntriesRPC. I believe that calling the application thread at these two places is appropriate because this is when either of these servers would find out that the commitIndex has changed. Calling it more frequently would be a waste of effort.
  • Client Interaction: Raft dictates that all requests must go to the leader, and in the real world, the client would not know which server the leader is. Thus, the client needs to guess and sends requests to a random server. In my design, the server either sends it a redirect response, or an ACK. My client has a timeout of 1 second before retrying the request to a new, random server as it assumes that the server it previously considered to be a leader had failed. My client does not move on to the next request until it receives an ACK for the previous, sending new requests in 1 second intervals. I chose a timeout of 1 second so that the receiver socket buffers would not be overwhelmed and drop data.
  • Key-Value Store: Due to the modularization of the code, the key-value store was relatively easy to add on. Each get, put, append command is embedded in its own log entry and replicated. When each server applies the commands, get commands are a no-op, puts and appends are performed in the corresponding key in my Python dictionary. The leader responds to get commands with the resulting values post-application of all previous commands, and ACKs puts and appends.

APAX

OCaml | Team Project

Nov 2017 - Dec 2017

Inspired by some seniors implementing Paxos, my friends and I, as part of an attempt to learn about distributed computing, decided to implement the Raft algorithm. Raft is a distributed consensus algorithm which allows clients to connect to any server in a server cluster and update a stored value. I was a backend developer in this project, and I worked on both leader election and log replication: I setup networking between servers and debugged leader-follower communications to automate leader election.* In addition, I designed structures (deciding how to store and process the RPC requests from clients, followers, and leader) for log replication for servers to reach a consensus, and tolerate failures within the clique.

*The leader is the server that the client is redirected to; all client requests must go through the leader, who sends a remote procedure call (RPC) to its follows and only commits the entry to its log if the update is accepted by a majority of the followers. The leader periodically sends heartbeat RPCs to its followers, and if a leader is down, the followers and supposed to automatically re-elect a new leader, so the system can resume processing client requests as normal.

Pawpulation

Cornell University Animal Health Hackathon | Java | Team Project

Jan 28, 2017 - Jan 29, 2017

In a nutshell, I worked with a team of vets, designers, and developers to enable real-time disease surveillance and research. I created a GUI that supports searching the database, and exporting and importing data.

The repository can be found here. Read more about my experience below...


My team originally comprised of my 3 developer friends and me. Our interdisciplinary team of 8, consisting of 3 vet school students and 1 Masters of Engineering student - all strangers - and us, came together because we were all drawn to the notion of a disease tracker for animals.

Currently, if a human patient goes into a hospital and is diagnosed with a disease, that data point is recorded and added to epidemiological databases. But in companion animal medicine in the United States, data collection is lacking. There are limited, accessible databases for some infectious diseases, reports on diseases are distributed infrequently, and for some diseases, data entry is even optional.

We envisioned a software that would streamline data entry (by parsing JSON) and upload data points to a database on a server. This database would update in real-time. That way, vets and researchers can easily search the database using criteria such as zip code and disease to track and prevent the spread of disease. Furthermore, the user can export data to CSV form, compatible, raw data for statistical analysis. Pharmaceutical companies, for example, could use this feature to develop more timely products and combat diseases quicker.

Our prototype allows data to be easily imported, exported, and searched. Unfortunately, we were not able to set up a database server to collect this data. Instead, the data imported was stored inside an ArrayList. Ideally, the data would be added to an SQL database for time-efficient and space-efficient retrieval. I mainly worked on the "database": adding data, filtering searches, returning a summary of data, and updating the client's dropdown boxes programmatically based on available data.


From this hackathon, I realized that engineering a product requires much much more than knowledge of programming languages and data structures. Engineering first requires a vision. For a majority of the time, I was extremely focused on coding to completing the product. In retrospect, a complete product is great, but only a well thought-out product will truly benefit people. That's why listening to the vet's discussions on feasibility (whether the data should be manually entered or import automatically), the business model (whether the market was the public, vets, or companies), etc. were so valuable for me. Although the hackathon was extremely draining (the developers stayed up from 10AM to 4AM the next day), I was motivated because of the vet students' passion. I originally participated in the hackathon for the experience - to challenge myself and see how quickly I could think and code in a pressurized-situation. I came out of the hackathon enlightened, understanding the importance of thinking about all aspects of a product and sincerely striving to create an impact with it, not just perfecting and optimizing the technical side.

My team is currently looking into ways this product can be developed further. Although unlike other hackathon projects, this project is not geared towards the general public as a market, this could be of great value to researchers. I hope that a revamped product of this kind can really be used to stop disease outbreaks and benefit animal and human lives.
